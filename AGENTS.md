# 仓库协作规范

本文件是仓库根目录下的默认协作约束，适用于本目录及其所有子目录。若更深层目录存在新的 `AGENTS.md`，以更近作用域的文件为准。

当前已知的局部覆盖规则：`apps/web/AGENTS.md` 会覆盖前端目录下的工作流、命令与验证要求；处理 `apps/web` 内文件时，必须优先遵循该文件。

## 1. 责任与基本原则

### 1.1 结果责任

1. 代理生成的代码、文档与命令结果，都按最终提交者的工程质量标准负责。
2. 不能以“这是模型生成的”为理由接受含糊实现、缺失错误处理、未验证结论或未同步文档。
3. 在结束一次改动前，必须能说明：改了什么、为什么这样改、如何验证它成立。

### 1.2 核心原则

1. 先保证长期可维护性，再追求短期堆功能。
2. 先抽象共享核心，再做 Web 界面或桌面壳。
3. 所有变更都必须留下同步更新的文档痕迹。
4. 任何阶段都要能明确回答：当前在做哪一步、下一步是什么、为什么还没做后一步。

### 1.3 当前项目方向

1. 当前阶段、当前细分步骤与下一步优先级以 `docs/status.md` 为唯一准确信源；需求边界与模块职责分别以 `docs/requirements.md`、`docs/architecture.md` 为准。
2. 当前阶段重点是“Web 界面 ↔ 运行时桥接收口”。
3. 当前优先事项是继续稳定共享层、收口服务桥接、巩固统一工具协议。
4. 下一优先方向是统一工具协议向外部协议映射与 MCP 接入，而不是继续堆厚客户端界面。
5. `apps/web` 是当前主客户端方向；`apps/agent-server` 是共享运行时的桥接层与控制面，而不是新的业务核心。

## 2. 设计哲学

### 2.1 Shared Core First

1. 新能力必须优先落入合适的共享 crate，而不是直接写进 `apps/agent-server` 或 `apps/web`。
2. 如果已有模块边界能够承载需求，优先扩展现有抽象，不新增平行机制。
3. 壳层负责桥接、组合与展示，不负责重新发明运行时、协议或持久化语义。

### 2.2 单一内部工具协议

1. 内部只维护一套统一工具定义与调用语义。
2. Claude、Codex、MCP 或其他外部协议的差异，只能在映射层解决，不能污染 `agent-core`、`agent-runtime` 或真实工具定义。
3. 模型可见工具名保持短名、稳定、与具体执行器解耦；不要把底层实现细节暴露成公共协议的一部分。

### 2.3 Web First，但不是 Web Only

1. 当前主界面承接层是 `apps/web`。
2. `apps/agent-server` 不只是“Web 后端”，它还是其他客户端可驱动的规范入口。
3. 桌面壳、终端入口、外部编排器应尽量复用同一运行时、同一事件流、同一会话磁带模型，而不是各自发展旁路协议。

## 3. 工作区边界

### 3.1 共享 crate 职责

- `crates/aia-config`
  - 只放应用级共享默认值、路径约定、稳定标识与构造 helper。
  - 不承载运行时编排、provider 业务、协议映射或算法阈值。

- `crates/agent-core`
  - 只放领域模型、消息/工具协议抽象、稳定公共类型。
  - 不依赖具体 provider、前端壳层或桥接协议细节。

- `crates/agent-core-macros`
  - 只放服务于共享核心抽象的宏能力。
  - 不放业务逻辑或应用壳层判断。

- `crates/agent-prompts`
  - 只放共享 prompt 资源、工具描述文本与提示装配逻辑。
  - 不夹带界面状态、路由流程或 provider 业务判断。

- `crates/session-tape`
  - 只放会话磁带、锚点、交接、重建逻辑。
  - 负责基于 `jsonl` 的会话快照落盘与重放索引载入。
  - 不承载界面渲染或 provider 适配。

- `crates/agent-runtime`
  - 负责编排模型、工具、会话与统一事件流。
  - 不承担具体界面渲染，也不承接外部协议桥接细节。

- `crates/builtin-tools`
  - 只放内建工具定义与执行逻辑。
  - 通过统一工具协议接入运行时，不承载外部协议映射。

- `crates/provider-registry`
  - 只放 provider 资料、活动项、落盘与载入逻辑。
  - 不反向污染核心抽象。

- `crates/channel-registry`
  - 只放 channel 静态配置、启停状态与本地持久化。
  - 不承担动态运行时桥接、会话映射或 provider 协议处理。

- `crates/openai-adapter`
  - 只放提供商适配与协议映射。
  - 不能把 provider 特有字段、协议概念或错误语义扩散回核心层。
  - 只有真正属于 OpenAI 及其兼容接口的映射才应放在这里；通用外部协议映射不要借道本 crate 落地。

- `crates/agent-store`
  - 只放本地 SQLite session/trace 存储与查询能力。
  - 不把路由、桥接、前端投影或平台协议逻辑塞入存储层。

### 3.2 应用壳职责

- `apps/agent-server`
  - 只做 Web / SSE / 外部 channel 与共享运行时之间的桥接。
  - 允许持有桥接层专属控制面与运行态，但不得演化成承载全部业务规则的巨石入口。
  - 不得在此重新实现一套 runtime 编排、工具定义层或 provider 专属业务规则。

- `apps/web`
  - 只做主界面承接、交互与展示。
  - 通过服务端桥接消费共享运行时能力，不重写 agent loop 或工具协议。

## 4. 代码规范

### 4.1 通用要求

1. 命名必须表达意图，避免含糊缩写。
2. 默认优先小而清晰的模块边界，避免把协议、运行时、存储、界面状态混在一起。
3. 修改前先阅读已有实现，优先复用现有抽象，不复制粘贴逻辑。
4. 没有必要时不要引入新依赖。
5. 错误必须显式处理，禁止静默吞错。
6. `lib.rs` 保持薄 façade，只承担模块声明与稳定导出，不堆放实现细节。
7. 发现重复逻辑、抽象漂移或边界泄漏时，可在与当前任务直接相关的局部范围内顺手收口到合理层级，但不要借机扩成无关重构。

### 4.2 Rust 约束

1. 生产路径禁止 `unwrap()`、`expect()`；测试中仅允许在断言上下文或真正不可能失败的搭建路径中谨慎使用。
2. 禁止用类型逃逸掩盖问题。
3. 优先返回结构化错误，而不是随意拼接字符串。
4. 公共类型、模块命名与构造入口要稳定，避免为短期实现频繁改名。
5. workspace lint 已显式拒绝以下模式，新增代码必须满足：
   - `unsafe_code = "forbid"`
   - `clippy::dbg_macro = "deny"`
   - `clippy::todo = "deny"`
   - `clippy::unwrap_used = "deny"`

### 4.3 前端约束

1. 处理 `apps/web` 内代码时，必须同时遵循本文件与 `apps/web/AGENTS.md`。
2. 前端只负责界面、交互与状态展示，不重写 Rust 运行时逻辑。
3. 前端工作流、格式化、测试与类型检查，优先以 `apps/web/package.json` 和 `apps/web/AGENTS.md` 的当前定义为准，不要想当然套用其他项目习惯。

## 5. 文档实时更新制度

任何会影响需求、边界、路线、阶段、当前步骤或下一步优先级的改动，都必须同步更新对应文档；若只是局部实现细节调整，也至少要显式检查这些文档是否仍然准确。

### 5.1 需求变更时

必须同时检查并更新：

- `README.md`：对外概览与项目方向
- `docs/requirements.md`：结构化需求与阶段边界

### 5.2 架构变更时

必须同时检查并更新：

- `docs/architecture.md`：模块职责、约束、协议方向
- 相关专题文档，例如：`docs/desktop-wrapper.md`、`docs/frontend-web-guidelines.md`、`docs/tool-schema-derive.md`

### 5.3 进度变化时

必须更新：

- `docs/status.md`

至少同步这些字段：

- 当前阶段
- 当前步骤
- 已完成事项
- 正在进行事项
- 下一步
- 当前阻塞（如果有）

### 5.4 自驱演进记录

若本次工作遵循 `docs/self.md` 所描述的自驱演进流程，或本次变更本身就是在收口演进轨迹，则应按既有格式追加更新 `docs/evolution-log.md`，而不是覆写历史记录。

## 6. 项目阶段维护规则

项目阶段不是口头描述，必须写进 `docs/status.md`。

建议使用以下阶段：

1. 愿景整理
2. 核心工作区搭建
3. 真实模型适配
4. 工具协议与 MCP 接入
5. Web 界面实现
6. 桌面壳接入
7. 稳定性与跨平台收尾

进入新阶段时，必须同时做三件事：

1. 更新 `docs/status.md`
2. 检查 `docs/requirements.md` 是否需要重划范围
3. 检查 `docs/architecture.md` 是否需要修正边界

## 7. 验证与格式化

完成代码改动后，必须运行与改动范围匹配的验证命令；不能以“理论上应该可用”代替实际验证。

### 7.1 Rust 相关改动

优先使用仓库已有入口：

```bash
just fmt
just check
just test
```

等价命令分别为：

```bash
cargo fmt --all
cargo check --workspace
cargo test --workspace
```

若本次改动范围更小，可以先跑受影响 crate 或更窄测试，再按需要扩大验证范围；但在结束前，必须明确说明最终验证覆盖到了什么。

### 7.2 前端相关改动

若涉及 `apps/web`，必须遵循 `apps/web/AGENTS.md` 中的本地规则，并以 `apps/web/package.json` 的当前脚本语义为准。通常可参考以下命令：

```bash
cd apps/web && ./node_modules/.bin/tsc --noEmit
cd apps/web && ./node_modules/.bin/vp test
```

若该目录当前脚本语义变动，以 `apps/web/package.json` 与局部 `AGENTS.md` 为准；不要机械把所有命令改写成单一入口。

### 7.3 联调命令

如需本地联调，可使用仓库已定义命令：

```bash
just dev
just dev-server
just dev-web
just typecheck
```

不要虚构仓库里不存在的 CI、发布或脚手架命令。

## 8. 提交与拒收规则

### 8.1 提交限制

1. 未经用户明确要求，不得执行 `git commit`、`git push`、创建标签或改写历史。
2. 未经用户明确要求，不得擅自创建分支、提交 PR 或修改远端状态。
3. 若用户要求提交，只提交与当前任务直接相关的改动，不夹带无关修复。

### 8.2 会被拒收的改动

以下模式会被视为低质量实现：

- 把共享能力直接堆进 `apps/agent-server` 或 `apps/web`
- 新增与现有抽象平行的第二套协议、第二套状态模型或第二套工具路径
- 生产路径 `unwrap()`、`expect()`、静默吞错或类型逃逸
- 不更新受影响文档，导致代码与项目叙事漂移
- 把 provider 细节、平台协议差异或壳层状态反向污染共享核心
- 没做实际验证，只给出“应该可以工作”的结论

## 9. 结束前检查清单

在结束一次非纯文本工作前，至少自检：

1. 代码是否遵守共享核心优先原则。
2. 需求文档是否需要更新。
3. 架构文档是否需要更新。
4. 状态文档是否准确反映当前步骤。
5. README 是否仍与当前仓库状态一致。
6. 是否能清楚说明为什么当前下一步优先是工具协议映射与 MCP 接入，而不是继续堆厚客户端界面。

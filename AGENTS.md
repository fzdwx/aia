# 项目协作规范

本文件定义本仓库中代理与人工协作者都必须遵守的最低约束。

## 1. 核心原则

1. 先保证长期可维护性，再追求短期堆功能。
2. 先抽象共享核心，再做 Web 界面或桌面壳。
3. 所有变更都必须留下同步更新的文档痕迹。
4. 任何阶段都要能明确回答：当前在做哪一步、下一步是什么、为什么还没做后一步。

## 2. 代码规范

### 2.1 通用要求

- 命名要表达意图，避免含糊缩写。
- 默认优先小而清晰的模块边界，避免把协议、运行时、界面状态混在一起。
- 变更时优先复用现有抽象，不要复制粘贴逻辑。
- 没有必要时不要引入新依赖。
- 错误必须显式处理，禁止静默吞错。
- 新能力优先放入共享库，而不是直接写进应用壳层。

### 2.2 Rust 约束

- 禁止 `unwrap()`、`expect()` 出现在生产路径；测试中允许在断言上下文使用。
- 禁止用类型逃逸掩盖问题。
- 优先返回结构化错误，而不是随意拼接字符串。
- 公共类型与模块命名要稳定，避免为短期实现频繁改名。
- 跨 crate 复用的应用级路径、默认值、稳定标识与构造 helper，优先收口到 `crates/aia-config`，不要在多个 crate / app 壳重复散落。

### 2.3 架构约束

- `aia-config` 只放应用级共享默认值、路径约定与稳定标识，不承载运行时编排、provider 业务、协议映射或算法阈值。
- `agent-core` 只放领域模型与协议抽象。
- `session-tape` 只放会话磁带、锚点、交接与重建逻辑。
- `session-tape` 还负责基于 jsonl 的会话快照落盘与重放索引载入。
- `agent-runtime` 负责编排模型、工具、会话，但不承担具体界面渲染。
- `agent-runtime` 产生统一事件流，界面层通过订阅消费事件，不得各自发明旁路输出协议。
- `provider-registry` 只放 provider 资料、活动项、落盘与载入逻辑。
- `openai-adapter` 只放提供商适配与协议映射，不得反向污染核心抽象。
- `apps/agent-server` 只做 Web ↔ 运行时桥接，不得演化成承载全部业务逻辑的巨石入口。

## 3. 文档实时更新制度

任何会影响需求、边界、路线、阶段的提交，都必须同步更新对应文档。

### 3.1 需求变更时

必须同时检查并更新：

- `README.md`：对外概览与项目方向
- `docs/requirements.md`：结构化需求与阶段边界

### 3.2 架构变更时

必须同时检查并更新：

- `docs/architecture.md`：模块职责、约束、协议方向
- 相关专题文档，例如：`docs/desktop-wrapper.md`、`docs/frontend-web-guidelines.md`

### 3.3 进度变化时

必须更新：

- `docs/status.md`

至少同步这些字段：

- 当前阶段
- 当前步骤
- 已完成事项
- 正在进行事项
- 下一步
- 当前阻塞（如果有）

## 4. 项目阶段维护规则

项目阶段不是口头描述，必须写进 `docs/status.md`。

建议使用以下阶段：

1. 愿景整理
2. 核心工作区搭建
3. 真实模型适配
4. 工具协议与 MCP 接入
5. Web 界面实现
6. 桌面壳接入
7. 稳定性与跨平台收尾

进入新阶段时，必须同时做三件事：

1. 更新 `docs/status.md`
2. 检查 `docs/requirements.md` 是否需要重划范围
3. 检查 `docs/architecture.md` 是否需要修正边界

## 5. 本项目当前状态

以 `docs/status.md` 为唯一准确信源。

当前要求：

- 必须把项目阶段显式标记为“核心工作区搭建”之后的当前细分步骤
- 必须写清楚已经完成了哪些基础骨架
- 必须写清楚下一步为什么优先是工具协议映射与 MCP 接入，而不是直接堆厚客户端界面

## 6. 执行检查清单

在结束一次非纯文本回答前，至少自检：

1. 代码是否遵守共享核心优先原则
2. 需求文档是否需要更新
3. 架构文档是否需要更新
4. 状态文档是否准确反映当前步骤
5. README 是否仍与当前仓库状态一致
